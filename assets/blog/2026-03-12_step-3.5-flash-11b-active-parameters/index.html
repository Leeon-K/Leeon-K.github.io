一句话总结

Step 3.5 Flash 是一个 196B 参数稀疏 MoE 模型，每 token 仅激活 11B 参数，通过 3:1 滑动窗口/全注意力混合架构 + MTP-3 推测解码，实现了 170 tokens/s 的推理速度，在数学、代码、Agent 任务上达到 frontier 级别性能，可比肩 GPT-5.2 和 Gemini 3.0 Pro。

论文摘要

我们推出 Step 3.5 Flash，这是一个稀疏 Mixture-of-Experts (MoE) 模型，桥接了 frontier 级别 agentic 智能与计算效率之间的差距。我们专注于构建 agent 时最重要的两个方面：敏锐的推理，以及快速可靠的执行。基于这些优先级，Step 3.5 Flash 配备了一个 196B 参数的基础模型用于高保真建模，以及 11B 激活参数用于高效推理，并通过交替的 3:1 滑动窗口/全注意力和多 Token 预测 (MTP-3) 优化，以最小化多轮 agentic 交互的延迟和成本。

为了达到 frontier 级智能，我们设计了一个可扩展的 RL 框架，集成可验证信号和偏好反馈，同时在大规模 off-policy 训练中保持稳定，以推动数学、代码和工具使用方面的持续自我改进。

Step 3.5 Flash 在 agent、coding 和 math 任务上展现出强大的智能，在 IMO-AnswerBench 上达到 85.4%，在 LiveCodeBench-v6 (2024.08–2025.05) 上达到 86.4%，在 τ²-Bench 上达到 88.2%，在 BrowseComp (w. Context Manage) 上达到 69.0%，在 Terminal-Bench 2.0 上达到 51.0%——性能与 GPT-5.2 xHigh 和 Gemini 3.0 Pro 等 frontier 模型相当。

研究背景与问题

核心挑战

  1. 效率 vs 智能的权衡：开源模型在复杂推理任务上仍落后于闭源 frontier 模型
  2. 长上下文 Agent 场景的延迟瓶颈：Agent 工作负载需要快速的 prefilling 和多轮交互解码
  3. MoE 训练稳定性：大规模稀疏 MoE 面临多种失败模式（专家崩溃、激活爆炸等）

设计哲学

Step 3.5 Flash 的架构反映了模型-系统协同设计的范式转变。除了传统的智能和成本目标外，自主 agent 时代提升了第三个关键约束：推理延迟。在交互式 agentic 工作流中，最小化延迟直接转化为减少任务完成时间，或通过测试时缩放在固定时间预算内增加智能。

Agent 工作负载具有独特的特征：大量 context prefilling 后是长时间的多轮交互解码。因此，我们沿三个耦合轴共同设计 Step 3.5 Flash 以实现低延迟：

  • 注意力：加速长上下文处理，与 MTP 良好兼容
  • 稀疏 MoE：防止分布式部署中的 stragglers 提高吞吐量
  • 多 Token 预测 (MTP)：通过推测解码促进快速生成

核心架构创新

1. 稀疏 MoE 主干

  • 45 层 Transformer：3 层 dense FFN + 42 层 MoE
  • 每层 288 个路由专家 + 1 个共享专家
  • Top-8 路由：每个 token 激活 8 个专家
  • 总参数 196B，激活 11B

2. 混合注意力布局（S3F1）

设计选择：3:1 Sliding Window Attention (SWA) : Full Attention

动机：

  • SWA 降低长上下文 prefilling 的 FLOPs
  • 保留全注意力层确保长程信息流通
  • 与 MTP 推测解码兼容性好（vs 线性注意力）

问题：Naive S3F1 性能下降明显

解决方案：

  1. 增加 SWA 的 Query Head（64→96）：几乎免费的午餐
  2. Head-wise Gated Attention：替代 sink token，作为"数据依赖的 sink"

3. 多 Token 预测 (MTP-3)

  • 添加 3 个轻量级 MTP heads（仅 0.81B 参数，0.41%）
  • 支持推测解码加速生成
  • 训练策略：仅 MTP-1 主训练，后期从 MTP-1 初始化 MTP-2/3 并联合微调

4. EP-Group Balanced MoE Routing

解决 Expert Parallelism (EP) 下的负载均衡问题：

  • 传统 loss-free balancing 只保证全局平衡
  • 引入 EP-level balancing loss，显式促进 rank 级利用率均匀

训练稳定性（重要贡献！）

训练稳定性是大规模稀疏 MoE 预训练的首要要求。通过轻量级异步指标服务器进行微批次级连续日志记录，构建全面的可观察性和诊断堆栈。该基础设施能够系统识别和缓解大规模 MoE 失败模式。

三大失败模式及解决方案

1. Muon 数值敏感性

  • 问题：Muon optimizer 在 bfloat16 下偶发极端 outlier
  • 解决：仅对 Polar Express 迭代使用 float16

2. 专家崩溃（Expert Collapse）

  • 新发现：专家崩溃可在路由统计稳定时发生（专家侧病理）
  • 根因：
    • 共享专家和路由专家的聚合需要显式缩放
    • 微批次 load balancing 过于严格
  • 监控：专家激活范数 + 参数范数的 min-to-median 比率

3. 局部激活爆炸（Localized Activation Blow-up）

  • 现象：深层 MoE 的少数专家激活范数爆炸，训练 loss 完全看不出
  • 解决方案：Activation clipping（而非 weight clipping）

关键洞察：训练 loss 完全无法反映这种内部不稳定性！必须监控 max-to-median 比率。

后训练：统一 RL 框架

核心挑战

  • 单一通用模型覆盖多领域 → 牺牲领域专业性
  • 维护多个专家模型 → 碎片化 + 高迭代成本
  • MoE + 长程推理 → off-policy rollouts 的梯度方差爆炸

MIS-PO (Metropolis Independence Sampling-Filtered Policy Optimization)

核心思想：用离散过滤替代连续 importance weighting

  • 在 token 和 trajectory 级别进行分布过滤
  • 限制在稳定 trust region 内优化
  • 大幅降低梯度方差，同时保留有效学习信号

贡献总结

  1. 精准的问题定义：抓住 Agent 场景的 latency 这一 third dimension
  2. 工程细节详实：训练稳定性诊断、负载均衡、监控基础设施
  3. S3F1+Head 设计：用"几乎免费"的 head 扩展弥补 SWA 性能损失
  4. MIS-PO 创新：解决 MoE + 长程 RL 的核心痛点
  5. 训练稳定性三大发现：Muon 数值敏感性、专家崩溃新形式、局部激活爆炸
